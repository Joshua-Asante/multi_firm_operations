# ADR: Striker v4.3 — pyramid 350% and T1 removed

**Date:** 2026-04-17
**Status:** Accepted
**Scope:** Striker DJ30 strategy (DJ30 15min long breakout)

## Context

Striker v4.1 used a T1/T2 partial exit structure with a pyramid-add mechanism. The partial structure was a standard breakout-strategy feature: take partial profit at T1 to reduce risk, then let the remainder run with pyramid additions.

MFE/MAE diagnostic on v4.1 CSV output produced a sharp finding: the pyramid cohort — trades where the pyramid add triggered — generated approximately 94% of total profit. The base cohort (no pyramid add) was near-zero contribution. T1 was firing before the pyramid structure had a chance to develop on many trades, locking in a small win and forfeiting the trade's actual edge.

The strategy's measurable edge was concentrated in pyramid-cohort extension moves. T1 was actively fighting that edge.

## Decision

Remove T1 entirely. Increase pyramid size to 350%. Accept higher per-trade variance in exchange for fully exploiting the edge concentration.

Additional parameter tightening accompanied the change:
- BE trigger: 0.15
- Trail trigger / wide: 0.15 / 0.9
- SL: 1.35 × ATR
- TP: 8 × ATR

## Alternatives considered

- **Keep T1 at lower threshold.** Rejected — the issue was not T1's threshold, it was T1's existence. Any T1 level truncates pyramid runners proportionally to how often it fires.
- **Keep T1, reduce pyramid size.** Rejected — this would have made the cohort problem worse by shrinking the profitable cohort further.
- **Remove T1, keep pyramid at v4.1 size (~150%).** Rejected on sensitivity grid — pyramid at 350% captured materially more of the edge without pushing DD beyond the 5% static bound.
- **Remove T1, remove pyramid, go flat-size.** Rejected — flat-size version underperformed because the measurable edge IS the pyramid cohort. Removing pyramid destroys what the strategy actually captures.
- **Full v5 rebuild (strip BE/trail/T1 following Guardian v5.1 pattern).** Not rejected — queued for post-challenge. Hypothesis: same pattern as Guardian v4 → v5.1. Not executed pre-challenge because the v4.3 refinement is a proven incremental improvement and the challenge is not the right forum for architectural experimentation.

## Consequences

Positive:
- 4yr backtest: Net +$422K, PF 2.55, RF 18.43, WR 72.2%, 237 trades, DD 5.59%. Pyramid cohort now captures its full edge.
- Simpler code path — T1 logic and its parameter surface removed entirely.
- The strategy's behavior now matches its measurable edge rather than fighting it.

Negative:
- μ/σ 1.24 remains the lowest in the portfolio (Guardian 1.54, Aegis 1.63). BE/trail are still present and are suspected of causing similar truncation that v5 rebuild would address.
- Individual trade outcomes more bimodal: clean pyramid runners vs full-stop losses. Psychologically harder than the prior T1-smoothed distribution.

Risks:
- **392-day max DD duration in backtest (N=1).** A single occurrence with no statistical basis for expectation. If a similarly long DD materializes live, there is no modeled playbook for how to respond. Monitor, do not react on DD duration alone.
- Regime dependence: pyramid edge assumes DJ30 breakout followthrough. In chop regimes (consolidation, range-bound markets), pyramid cohort size shrinks and base cohort drives results — but base cohort is near-zero contribution. Extended chop regime would compress returns substantially.
- DXTrade `contractValue` must be **10** to match backtest assumptions. Default of 1 produces approximately 7% per-trade risk, catastrophically over-sized. See operational rules.
